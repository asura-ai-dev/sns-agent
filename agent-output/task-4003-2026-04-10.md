## known_gaps

- Repository の `aggregate` は `UsageRepository` インターフェース上「期間全体を platform でまとめた集計」のみを返す。`getUsageReport` は period バケットに分割して `aggregate` を複数回呼ぶ実装としており、period 数が多い range（たとえば daily で 1 年分 = 365 クエリ）ではクエリ数が膨らむ可能性がある。クエリ数上限は guardrail（daily 366, weekly 60, monthly 36）で抑えているが、本格的な SQL ベース期間バケット集計は v1.5 以降で検討する。
- `middleware/usage-recorder.ts` は context 注入に留めており、HTTP 層で自動的に使用量を記録する機能は持たない。実際の使用量記録は `packages/core/src/usecases/post.ts` の `recordProviderUsage` ヘルパー経由でユースケース層から行う設計（design.md 4.4, architecture.md 13 の方針）。
- スケジュールジョブ経由の発行（`schedule.ts` → `publishPost`）も同じ `postUsecaseDeps` 経由で `usageRepo` を受け取り記録されるため Job 経由の使用量も捕捉されるが、Schedule 自体のスケジューリング行為は usage 対象外とした。
- LLM 呼び出し側（`packages/llm`）からの使用量記録連携はまだ行っていない。LLM 経路の usage 記録は Phase 5 の agent gateway 実装で結合される想定。
- `/api/usage` のレポートは Repository から取ったまま返しており、空バケット（件数 0 の日）は結果に含めない。UI でグラフを描く際は欠損日のゼロ埋めを UI 側で行う必要がある。
