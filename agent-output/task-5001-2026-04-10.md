## 既知のギャップ / known_gaps

- **実 LLM 呼び出しのライブテスト未実施**: openai/@anthropic-ai/sdk の実 API 叩きテストは API キーが必要なため未実行。アダプタのエラー変換・ストリーム処理はモックベースで検証済み。統合テストは Phase 5 後半の agent/chat エンドポイント実装時に行うのが妥当。
- **ストリーミング経路の実地検証なし**: `stream()` は型と分岐のみ検証。SSE 統合は Task 5003+（agent gateway）側で併せて確認する必要あり。
- **API キー取得は未実装**: `buildDefaultAdapters` は呼び出し側が API キーを渡す前提。環境変数 `OPENAI_API_KEY` / `ANTHROPIC_API_KEY` の読み取りは本ティケットのスコープ外（config/env.ts のスキーマ拡張も未実施）。executeLlmCall を使う API エンドポイント実装時に追加する。
- **llm:read は現状 admin+ のみ**: rbac.ts の rolePermissions で admin/owner にのみ付与されている。design.md 4.2 の要件は「admin 以上」で満たしている。Phase 6 で LLM 使用量の viewer 閲覧が欲しくなったら要拡張。
- **llm-route-repository.resolve() の互換変更**: 旧 SQL ベース実装を specificity スコアリングに置き換えた。呼び出し側は LlmRouteRepository interface のまま変わらないが、DB アクセスが「全件取得+アプリ層フィルタ」になっている。ワークスペースあたり数十件を想定しており性能問題は無いが、膨大になった際は SQL への push down を再検討する。

## 重要判断

- **Repository 層 vs Router 層の解決ロジック重複**: `DrizzleLlmRouteRepository.resolve()` と `resolveLlmRoute()` がほぼ同じスコアリングを持つ形になった。router 側は Repository の `findByWorkspace` を叩くため、Repository の `resolve()` を直接使っていない。これは Repository interface として「resolve は single-query API」を保ちつつ、アプリ層で柔軟に DI テストできるようにするための冗長性。将来的に統一する場合は router 側で repo.resolve() を使う形にリファクタできる。
- **recordUsage 失敗時の握り潰し**: LLM 呼び出しの正常系を壊さないことを優先し、console.error のみとした。audit 層と違って usage_records の欠損は致命的でないという v1 判断。必要なら observability 層（Phase 6）で retry queue を入れる余地あり。
- **agent の actor としての usage 記録**: `ExecuteLlmDeps.actor` に `workspaceId/actorId/actorType` を外部から注入させる設計。これにより Agent Gateway 側で `actor.actorType = 'agent'` を渡すだけで課金主体が分離される。
