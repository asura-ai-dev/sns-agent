## 完了済み項目

### 1. core 投稿ユースケース (`packages/core/src/usecases/post.ts`)

- `createPost(deps, input)`
  - `idempotencyKey` 指定時は `postRepo.findByIdempotencyKey` で既存投稿チェック
    - 同一ワークスペースなら既存 Post を返す（冪等）
    - 別ワークスペースのキー衝突は `ValidationError`
  - アカウント取得 + ワークスペース一致確認 + `status === 'active'` チェック
  - `provider.validatePost()` を呼び、`validation_result` を生成
  - 失敗時は `ValidationError` を throw し DB には保存しない
  - 成功時は `status='draft'` で保存、`validationResult` も永続化
  - `publishNow: true` の場合は `publishPost` を連鎖呼び出し
- `updatePost(deps, workspaceId, postId, input)`
  - 下書きのみ編集可能、`status !== 'draft'` は `ValidationError`
  - 更新内容で再バリデーションし、`validationResult` を更新
- `publishPost(deps, workspaceId, postId)`
  - `status` を一旦 `publishing` に遷移させてから Provider 呼び出し
  - `credentialsEncrypted` を `decrypt` してから `provider.publishPost` に渡す
  - 成功: `status='published'`, `platformPostId`, `publishedAt` を更新
  - 失敗 (result.success=false or throw): `status='failed'` にし `validationResult.publishError` を追記、`ProviderError` を throw
- `deletePost(deps, workspaceId, postId)`
  - draft / publishing / scheduled / failed: 直接 `status='deleted'` に更新
  - `published` かつ `platformPostId` あり: `provider.deletePost` を呼び、成功後に `status='deleted'`
  - 既に `deleted` なら冪等に同じ Post を返す
- `listPosts(deps, workspaceId, filters)`
  - filters: `platform`, `status`, `from`, `to`, `page`, `limit`
  - `from`/`to` は `createdAt` でアプリ層フィルタ（リポジトリは platform/status のみ対応）
  - `page`, `limit` でページング。デフォルト page=1, limit=20, 最大 limit=200
  - `{ data: Post[], meta: { page, limit, total } }` を返す
- `getPost(deps, workspaceId, postId)`: ワークスペース所有権チェック付きの詳細取得

エクスポートは `packages/core/src/index.ts` に追加済み（`createPost`, `updatePost`, `publishPost`, `deletePost`, `listPosts`, `getPost`, 型 `PostUsecaseDeps`, `CreatePostInput`, `UpdatePostInput`, `ListPostsFilters`, `ListPostsResult`）。

### 2. Idempotency ミドルウェア (`apps/api/src/middleware/idempotency.ts`)

- `X-Idempotency-Key` ヘッダを検出（大文字小文字不問）
- POST / PATCH のみ対象、GET / DELETE / その他はスルー
- `/api/posts` 配下では DB の `posts.idempotency_key` を `DrizzlePostRepository.findByIdempotencyKey` で lookup
  - ヒットかつワークスペース一致なら 200 で `{ data: Post }` を返しハンドラをスキップ
  - 非ヒット時は `c.set('idempotencyKey', key)` で下流ハンドラへ伝搬
- `AppVariables` に `idempotencyKey?: string` を追加
- `apps/api/src/middleware/index.ts` から `idempotencyMiddleware` を再エクスポート

### 3. 投稿 API (`apps/api/src/routes/posts.ts`)

既存のスタブを完全置換。ルートレベルで `idempotencyMiddleware` を適用:

| メソッド | パス                   | 権限         |
| -------- | ---------------------- | ------------ |
| GET      | /api/posts             | post:read    |
| POST     | /api/posts             | post:create  |
| GET      | /api/posts/:id         | post:read    |
| PATCH    | /api/posts/:id         | post:create  |
| DELETE   | /api/posts/:id         | post:publish |
| POST     | /api/posts/:id/publish | post:publish |

- `serializePost()` で `validationResult` を含む JSON 整形
- `POST /api/posts` body: `{ socialAccountId, contentText, contentMedia?, publishNow?, idempotencyKey? }`
  - ヘッダ `X-Idempotency-Key` があれば優先（body 指定があれば body が上書き）
  - 成功時 201
- `GET /api/posts` クエリ: `platform`, `status`, `from`, `to`, `page`, `limit`
  - 不正な platform / status は `ValidationError` (400)
  - レスポンス: `{ data: Post[], meta: { page, limit, total } }`
- `buildDeps()` は `DrizzlePostRepository`, `DrizzleAccountRepository`, `ProviderRegistry`, `ENCRYPTION_KEY` を注入
  - `ENCRYPTION_KEY` 未設定時は開発用デフォルト鍵にフォールバック（accounts.ts と同方針）

### 4. Post レスポンス形式

- 一覧: `{ data, meta: { page, limit, total } }`
- 詳細 / 作成 / 更新 / 公開 / 削除: `{ data }`
- `data` 内に `validationResult` を必ず含める（UI でエラー表示に利用可能）

- `idempotency` v1 は `posts.idempotency_key` ユニーク制約ベース。汎用 idempotency テーブル（リクエストハッシュ + レスポンスキャッシュ）は未導入。設計書でも v1 最低限の方針。
- 作成リクエストで `publishNow: true` の場合、`post:publish` 権限を別途チェックしていない（ルートは `post:create` のみ）。editor 以上は post:create と post:publish の両方を持つので現状は問題なし。operator は post:create は持つが post:publish は持たないため、publishNow=true で失敗させる追加チェックは将来対応の余地あり。

## 既知のギャップ

- scheduled_jobs 連携（Task 2005）: 本タスクでは予約を扱わないため `createPost` は `publishNow` の 2 択のみ。予約作成は別チケットで scheduled_jobs と結線される前提。
- `DrizzlePostRepository.findByWorkspace` は `limit` / `offset` を引数で受け取るが、日付範囲フィルタを持たない。listPosts は安全側に全件取得 + 絞り込みに寄せたため、現状 DB 側の index（`idx_posts_workspace_platform_created`）は活かしきれていない。
- `idempotencyMiddleware` は `/api/posts` 以外では単にヘッダ値を context にセットするのみ。他リソースの idempotency 対応は各ルートで個別に実装する必要あり。
- publishNow=true で operator ロール（post:publish 権限なし）が叩いた場合、ルートミドルウェアは通過するが実際には下書き保存 + 即時公開が走ってしまう。権限不整合の追加チェックは次イテレーションで post:publish チェックを入れる候補。

## 重要判断

1. **publishPost の失敗処理**: Provider が throw した場合も `status='failed'` に遷移させてから ProviderError を上位に投げる。呼び出し側（API / UI）は 502 レスポンスを受け取りつつ、DB には失敗状態が永続化される。これにより評価観点「主要機能が動かない / 状態遷移が正しく永続化される」を満たす。
2. **validationResult の永続化**: `provider.validatePost` の結果をそのまま `posts.validation_result` に保存する。更新時も再バリデーションを行い上書きする。API レスポンスでも `validationResult` フィールドとして露出し、UI で警告・エラー表示に使える。
3. **Idempotency 実装の最小化**: 設計書の「v1 では DB の `idempotency_key` カラムでの重複検出を最低限実装する」に準拠し、汎用キャッシュ層は作らない。`/api/posts` に特化したミドルウェアとして、`DrizzlePostRepository.findByIdempotencyKey` を直接叩く実装を選択。
4. **日付フィルタの場所**: リポジトリインターフェースを変更せず、ユースケース層で `createdAt` を JS フィルタ。repository のインターフェース変更は影響範囲が広いため見送り、v1 スコープ優先。
5. **権限設計**: DELETE と POST /publish は `post:publish`（editor 以上）、GET は `post:read`、POST/PATCH は `post:create`。task 指示通り。operator が publishNow=true を使うとルートは通過してしまう件は既知ギャップとして残した。
