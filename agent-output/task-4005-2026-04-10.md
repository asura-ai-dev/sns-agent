  - クライアント側 CRUD UI
  - 既存ポリシー一覧テーブル（scope / period / limit / on exceed / 消費バー / edit・delete ボタン）

## known_gaps

- ブラウザ実機での視覚確認（`agent-browser`）未実施。デザインは既存 Operations Ledger の
  パターンに忠実に揃えているが、recharts の実レンダリング結果と余白の最終調整は
  evaluator フェーズで目視確認が必要
- `BudgetPolicyManager` のクライアント側 `fetch` は `NEXT_PUBLIC_SNS_AGENT_API_URL` を
  参照し、未設定時は `http://localhost:3001` に固定。本番で Next.js rewrite を使う場合は
  環境変数で上書きする必要がある（既存の accounts 画面と同じ前提）
- RBAC：ページレベルでの viewer ガードはサーバー側ミドルウェア（Task 4004 の `budget:manage`）
  が担当するため、UI ではボタン disable のみ。viewer ロールで /settings/budget を直接開いた
  場合の UI 表示制御は未実装（既存 accounts 画面と同等水準）
- 前期比（前期 cost 差分）はプラットフォームフィルタ切り替え時も全体の previous totals を使用。
  厳密な platform 別 delta にはサーバー側集計の拡張が必要
- usage 画面の LCP: recharts を動的 import せず同期 import しているため First Load JS が
  138 kB。パフォーマンス目標を下回る場合は `next/dynamic({ ssr: false })` 検討の余地あり
- ユニットテスト: `BudgetPolicyManager` のフォームバリデーション・API エラーハンドリングを
  カバーするテストが未整備
