## known_gaps

- `publishPost` でのメディアアップロードは **media_id 既出前提**。X v2 の media upload（v1.1 の `POST media/upload` 経由）は v1 スコープ外として未実装。contentMedia の `url` が外部 URL ではなく media_id 文字列として渡される運用を `parseCredentials` / `publishPost` のロジックでサポートしている。本格運用には別途 media upload フローが必要（Phase 4 の UsageApi 対応と合わせて検討）。
- `InMemoryVerifierStore` は単一プロセス用。マルチインスタンス配置時は DB / Redis 等に差し替え必要（Phase 4 / deploy 時）。
- `SocialProvider.refreshToken(accountId)` の契約は「accountId 文字列を受ける」が、v1 の XProvider は credentials JSON または生の refresh_token を受ける拡張解釈にしている。usecase 層（`packages/core/src/usecases/account.ts`）側で復号した credentials を渡す呼び出し規約を明示する必要（Phase 2 の後続タスクで調整可）。
- X API v2 スレッド投稿は `in_reply_to_tweet_id` で実現できるが、チケット仕様に合わせ `threadPost: false` としており未実装。`commentReply: true` に対応する返信 API は `sendReply` 実装時（Phase 6 inbox）に追加予定。

## 重要判断

- **X API クライアントは fetch ベース + FetchLike 差し替え**：テスト容易性と Node 20+ のネイティブ fetch 採用。axios / undici 依存を増やさない方針。
- **レート制限は throw ではなく構造化エラーで返す**：`RateLimitError`（code: `PROVIDER_RATE_LIMIT`）を `ProviderError` の派生として返すことで、上位で resetAt に基づくリトライ戦略を組めるようにした（DomainError の `details` に rateLimit を含めて伝搬）。
- **PKCE verifier の保存**：ProviderRegistry に state を引き回さず、`VerifierStore` 抽象を provider-x 内部に閉じた。v1 は in-memory、将来 DB/Redis へ差し替え可能。
- **credentials は provider-x 内部で JSON string 前提**：`accountCredentials: string` を `{ accessToken, refreshToken, expiresAt, scope, tokenType }` の JSON として取り扱う。core の `SocialAccount.credentialsEncrypted` は usecase 層で暗号化／復号しており、provider-x はプレーンテキスト JSON を受け取る契約。
- **`apps/api` 側のレジストリはシングルトン**：`getProviderRegistry()` で cache。`resetProviderRegistry()` でテスト用リセット可。ルート側は `Map` を `AccountUsecaseDeps.providers` にそのまま渡せる (`registry.getAll()` は内部 Map を返す設計)。
