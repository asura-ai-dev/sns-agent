## 重要判断

1. **クライアント側 fetch の配置**: 既存 `BudgetPolicyManager.tsx` が自前の `getApiBase()` /
   `apiFetch()` ヘルパを持っていた。SDK の llm/skills リソース化は影響範囲が大きいので
   同じ流儀を踏襲し、`LlmRouteManager` と `SkillsManager` の中にローカルな `apiFetch`
   を置いた。一方サーバー側 safe fetcher は `apps/web/src/lib/api.ts` にまとめ、指示通り
   「未起動時 fallback 対応」の一元化地点を守った。
2. **Temperature スライダーの UX**: spec は単に「スライダー」とのみ指定。Editorial トーンと
   整合させるため、紙面風パネル内に Fraunces で現在値を大きく表示し、tick ラベル（0.0〜2.0）
   を mono uppercase で配置する「ダイヤル」的な見せ方にした。
3. **Skills 詳細モーダルでの manifest 取得**: 一覧 API (`GET /api/skills`) は manifest を
   返さず `actionCount` のみ返す設計だったので、詳細モーダル起動時に
   `GET /api/skills/:id/manifest` を遅延 fetch する実装とした。ローディング / エラー状態
   を内部で持つ。
4. **`/skills` ページの masthead**: `/skills` は `(dashboard)/layout.tsx` 直下の top-level
   画面で `SettingsShell` 配下ではない。Settings 系のみに閉じたヘッダではなく、共通の
   ダッシュボードレイアウトに違和感なく馴染むよう、`SettingsShell` のヘッダ部と同じ装飾を持つ
   専用 `SkillsMasthead` を切り出した。
5. **Platform 'common'**: `SkillsManager` では skills 側の `common` プラットフォーム値が
   `PlatformIcon` でサポートされない（PlatformIcon は x/line/instagram のみ）ため、
   dashed border の `*` プレースホルダで代替表示した。LLM の `null` platform（デフォルト）も
   dashed `—` で区別している。
