    投稿制約定数 (TEXT_LIMIT=2200 / MAX_IMAGES=10 / MAX_VIDEOS=1)

    `X-App-Usage` / `X-Business-Use-Case-Usage` ヘッダを解析して rate limit 情報を返し、
    429 / code=4,17,32,613 を `RateLimitError` に変換、その他 non-2xx を `ProviderError` に変換。
  - `auth.ts`: Facebook Login 経由の OAuth 実装。
    - `getAuthUrl` (facebook.com/v19.0/dialog/oauth)
    - `exchangeCode` (short-lived user token)
    - `exchangeForLongLivedToken` (fb_exchange_token grant で 60 日 token)
    - `refreshLongLivedToken` (同エンドポイントで延長)
    - `fetchInstagramBusinessAccount` (/me/accounts から IG Business Account ID / username を解決)
  - `post.ts`: validatePost / publishPost / deletePost。
    - validatePost: 画像必須 (メディア 0 件で error)、キャプション 2200 文字 (コードポイント単位)、
      画像 10 枚上限、動画 1 本上限、非 HTTPS URL に warning
    - publishPost: 単一画像 / 単一動画 (media_type=REELS) / カルーセル (children container →
      parent container) の 3 パスを自動判別。いずれも Container 作成 → `/media_publish` の
      2 ステップ非同期。
    - deletePost: `DELETE /{ig-media-id}` を呼び、`204` または `{ success: true }` で成功判定。
  - `inbox.ts`: Messenger Conversations API (`platform=instagram`) でスレッド/メッセージを取得。
    参加者 ID から direction (`inbound`/`outbound`) を判定。ページングは `paging.cursors.after`。
  - `webhook.ts`: `X-Hub-Signature-256` の HMAC-SHA256 検証 (timingSafeEqual)。
    messaging (DM) / changes.comments をそれぞれ `WebhookEvent{type:"message"/"postback"}` に変換。
  - `index.ts`: `InstagramProvider` クラス。全 SocialProvider 必須メソッド + listThreads /
    getMessages / handleWebhook / refreshToken を実装。
- **package.json**: `@sns-agent/core` / `@sns-agent/config` を `workspace:*` で追加、

## 既知のギャップ (known_gaps)

- **リアルな Graph API 呼び出しは未検証**: 全テストは fetch モックによる単体テストのみ。
  実際の Facebook App / Instagram Business Account で E2E 検証するには
  `INSTAGRAM_CLIENT_ID` / `INSTAGRAM_CLIENT_SECRET` と IG Business Account が必要。
- **動画アップロード非同期ポーリング未実装**: Instagram Graph API は reels / video の
  Container 作成後 `status_code=FINISHED` になるまで数秒〜数十秒かかる場合があるが、
  本実装は作成直後に `media_publish` を呼ぶ。長時間エンコードが必要な動画では 400 が返る
  可能性があり、将来的に `GET /{container-id}?fields=status_code` でポーリングする機能を
  追加する必要がある。v1 の仕様 (design.md セクション 11.1) は「リール対応」までで、
  ステータスポーリングは明示されていない。
- **inbox の attachments 未対応**: 画像/動画メッセージは `contentMedia: null` として
  返す (v1 はテキスト本文のみ)。
- **Webhook イベントの enqueue 未実装**: 受信したイベントは受理数を返すのみで、
  DB 保存や job queue への積み込みは行っていない (apps/api 側の後続タスクで実装予定)。
- **X の webhook エンドポイントは 501 のまま**: 本タスクのスコープ外。
- **LineProvider の registration ブロックが providers.ts に未追加**: import のみ存在し
  registry.register 呼び出しは無い。これは Task 4001 の責務で本タスク外。
  TypeScript は noUnusedLocals を設定していないためビルドは通る。

## 重要判断

- **credentials 形式**: `{ accessToken, igUserId, pageAccessToken, expiresAt, tokenType, refreshToken: null }`
  の JSON を採用した。Instagram は refresh_token を返さないため `refreshToken: null` 固定。
  usecase 層で暗号化する前提は provider-x と同じ。
- **pageAccessToken と User Access Token の使い分け**: `fetchInstagramBusinessAccount` で
  `/me/accounts` から取得した Page Access Token を credentials に含めているが、publishPost /
  deletePost / messenger conversations は primary の `accessToken` (long-lived user token)
  で呼んでいる。v1 はこれで動作する想定 (Facebook App が advanced access を持つ前提)。
  Page-scoped token が必要なら呼び出し側 (usecase) で差し替える拡張余地を残している。
- **Container → media_publish の 2 ステップ**: provider-x の 1 コール publish と異なる非同期 API
  形状を、provider 内部で隠蔽。呼び出し側は publishPost 1 回で完了する。
- **メディア URL は呼び出し元が公開 HTTPS URL を渡す前提**: Instagram Graph API は
  image_url / video_url が公開アクセス可能な HTTPS URL である必要がある。v1 は media upload
  機能は provider 外で行い、HTTPS URL を `MediaAttachment.url` で渡す運用とした (provider-x と同様)。
- **webhook の raw body 扱い**: apps/api 側で `c.req.text()` を呼んでから provider に渡す
  ことで、Hono の JSON パース前の生文字列に対して HMAC を計算するようにした。ミドルウェア
  チェーンで body が消費される前に webhook ルートへ到達する必要があるが、現状の app.ts は
  webhooks ルートに authMiddleware を掛けていないため問題なし。
