  - `SkillAction`: name, description, parameters (JSON Schema), permissions[], requiredCapabilities[], readOnly?
  - `SkillJsonSchema`: type / properties / required / items / enum / minimum / maximum / minLength / maxLength / pattern / additionalProperties をサポートする最小サブセット
  - `SkillPlatform`: "x" | "line" | "instagram" | "common"
  - `validateSkillManifest(input)` バリデーション関数
    - name は `/^[a-z0-9][a-z0-9_-]{1,63}$/`
    - version は semver 互換パターン
    - platform の enum 検証
    - actions は 1 件以上、name は重複不可、action 名は `/^[a-z0-9][a-z0-9._-]{0,95}$/`

## known_gaps

- **placeholderInvoker**: `apps/api/src/routes/agent.ts` 内の `SkillActionInvoker` は v1 時点ではダミーで、`{ status: "deferred", message: "Skill action mapping is not yet implemented" }` を返すのみ。実際の core use case (createPost / schedulePost / publishPost 等) への結線は Task 5003 以降 (skills パッケージ生成機構) で skill action 名 → use case マッピングを定義する段階で行う予定。Task 5002 のスコープは「実行基盤 (manifest + executor + gateway + API)」で、各 SNS 専用 skill のアクション実装は別チケット。
- **SSE ストリーミング未実装**: design.md では Web UI チャットは SSE 前提だが、v1 初期は非ストリーミング JSON 応答で返している。`/api/agent/chat` の応答形式は `{ data: { kind: "text" | "preview", ... } }` の単発 JSON。SSE 化は Web UI チャット画面 (Task 3008 予定) と合わせて次イテレーションで追加する。
- **会話 ID の永続化なし**: conversationId はクライアントから受け取るだけで、専用テーブルは用意していない。履歴参照は audit_logs の resourceId 経由で疑似的に実現している。専用の `conversations` テーブルは Phase 6 の承認フローと合わせて追加検討。
- **LLM キー環境変数経由**: `process.env.OPENAI_API_KEY` / `process.env.ANTHROPIC_API_KEY` を直接参照している。将来的に workspace_settings テーブルへ移す (design.md 1.8) が、Task 5002 では env 経由のまま。
- **action 名の実行モード 'draft' 解釈**: draft モードでは「書き込み系 skill アクションでも publishNow=false を強制する」という仕様だが、executor はこの強制を invoker 側に委ねている。実際の post.create skill action 実装時に draft モードを受けて `publishNow=false` で core.createPost を呼ぶマッピングを書く。
- **DrizzleSkillPackageRepository のテスト未整備**: db パッケージは既存方針 (test 未整備) に合わせて単体テストを追加していない。
- **audit_logs フィルタの粒度**: `/api/agent/history` は `actorId` + アプリ層フィルタで agent 系 action のみ抽出している。audit_log_repository に action の集合フィルタ (`action IN ...`) が無いため、取得後に filter している。大量データ時はページングが崩れるので将来的にリポジトリ側でサポートする。

## 重要判断

- **循環依存回避のため core は llm / skills を import しない**: Agent Gateway ユースケース (core) は `AgentLlmInvoker` / `AgentDryRunInvoker` / `AgentExecuteInvoker` を関数型の DI として受け取る。実装は apps/api 側で packages/llm + packages/skills を組み合わせて渡す。これにより core は純粋なドメイン層を維持し、将来 LLM/skills 実装を差し替え可能にする。
- **skill 実行結果は v1 では placeholder**: invoker mapping は Task 5002 のスコープ外 (manifest + 実行基盤が責務)。Task 5002 の goal は「manifest 駆動で検証・認可・dry-run・監査まで一気通貫で動く経路」の確立であり、実装検証は `{ status: "deferred" }` を返すダミー invoker で行った。本番マッピングは Task 5003 で実装する。
- **SkillPackageRepository の新設**: Agent Gateway が skill_packages を読む経路が必要なため、既存の repository パターンに沿って `DrizzleSkillPackageRepository` を追加。これにより apps/api/routes/agent.ts は drizzle 直接 import を避けて core interface に依存する形に整理された。
- **read-only モードは action.readOnly=true のみ許可**: spec にある read-only 実行モードの意味を「action 側でメタデータ `readOnly` を宣言したものだけ通す」と解釈。これにより draft / approval-required モードとの違いが明確化し、manifest 作成者が安全に読み取り系だけを識別できる。
