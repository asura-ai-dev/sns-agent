## 実装概要

軽量な自前カレンダー実装。外部カレンダーライブラリ（fullcalendar 等）は導入せず、6 週 × 7 日の月グリッドと月曜始まり 7 日タイムラインを素の React + Tailwind + DaisyUI + @phosphor-icons/react で構築した。既存の inbox / approvals 画面のエディトリアル質感（warm off-white base, Fraunces 見出し italic, 細罫線, 控えめなモーション）と統一している。

## known_gaps

- `ScheduleDetailModal` の reschedule / cancel を実 API 相手に endtoend で踏んでいない（API 側は `schedules.ts` で PATCH/DELETE が実装済みなので、実起動環境で最終確認が必要）。
- 月ビューのセル内チップは「時刻順に上から 2 件 + overflow カウンタ」のみで、同時刻の重なり可視化は未対応。v1 スコープの「ドット/小カード表示」範囲内と判断。
- 週ビューの予約カードは固定高（48px / 1h 相当）で配置しており、複数予約の overlap で視覚的に重なる可能性がある。v1 スコープ（タイムライン表示）として許容している。
- `/posts/new?scheduledAt=...` の受け取り実装は task 3004（投稿作成画面）側の責務。本タスクでは push するだけ。
- モバイル（375px）は現状 MonthView のセル高が 7rem 固定で横スクロール気味になる可能性がある。デスクトップ 1280px では問題なし。AC-15 のモバイル対応は MonthView のレスポンシブ縮約を Phase 3 後半で詰める余地あり。
- 自動テスト（vitest/playwright）は apps/web にまだ存在しない（既存 package.json で `"test": "echo \"No tests yet\""`）ため、このタスクでも追加していない。

## 重要判断

- **ライブラリ**: `@fullcalendar/react` 導入を見送り、軽量自前実装を採用。理由: (1) タスク指示が「軽量実装推奨」と明記、(2) 既存 inbox などと統一したエディトリアル質感を出すには自前の方が自由度が高い、(3) bundle size 圧縮（/calendar First Load JS 119 kB に収まった）。
- **aesthetic**: 既存の Warm off-white + Fraunces italic + 細罫線ベースのエディトリアル質感に寄せた。新しいビジュアル言語を導入せず、Almanac（暦）というテーマで統一感を維持している。
- **週始まり**: 月曜始まりで統一。日本の業務カレンダー慣習に合わせ、土曜を info tint、日曜を error tint で示す。
- **fallback 設計**: API 404/ネットワーク失敗時に `entries=[]` にして警告バナーだけ出す設計。これにより Phase 3 の他画面を触っている開発者が API オフラインでもカレンダー UI を確認できる。
- **新規予約導線**: 日付クリック・スロットクリック・「新規予約」ボタンの 3 経路すべてで `/posts/new?scheduledAt=...` にクエリ渡ししている。投稿作成画面側が未対応でも query param を無視すれば普通に開けるので疎結合。
