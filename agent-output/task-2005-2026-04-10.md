## 設計上の判断

1. **cancel の status**: design の JobStatus に `canceled` がないため、`failed + lastError='canceled_by_user'` でマークする運用にした。DELETE はベキ等ではなく、succeeded / running / locked 状態はエラーとして弾く。
2. **executeJob が Post を draft に一時遷移**: `publishPost` が `status === 'draft'` を要求する既存仕様のため、executeJob 内で Post を scheduled → draft に更新してから publishPost を呼ぶ。publishPost 内部でさらに `publishing → published/failed` に遷移する。
3. **lockJob の拡張**: 従来 `pending → locked` のみだったが、retrying 状態からも lock 取得できるよう `inArray(status, ['pending','retrying'])` に変更。stale locked 状態の再取得はワーカー側 findExecutable で対象化し、通常の lockJob で再取得する設計（tick の対象に含まれれば UPDATE は行われる）。※ ただし 現在の lockJob は pending/retrying のみを許すため、stale locked の再取得は「locked のまま executeJob を呼ぶと null」になる。フォローアップ欄参照。
4. **findByWorkspace を Repository に追加**: core の interface には追加せず、Drizzle 実装の拡張メソッドとして持たせ、usecase 側は `as (Repository & { findByWorkspace?: ... })` で optional 参照する。core interface の大規模変更を避けるため。findPendingJobs をフォールバックに持つ。
5. **auditMiddleware / authMiddleware の配線**: すでに app.ts で `/api/schedules/*` に設定済みだったため追加不要。

## Known Gaps / フォローアップ

- **stale locked の再取得完全性**: `findExecutable` では stale locked を返すが、`lockJob` は pending/retrying 以外を許さないため、stale locked ジョブに対する再取得が成立しない。現状はデッドロックからの回復には DB 直操作か追加メソッドが必要。次イテレーションで `recoverStaleLock(id, before: Date)` を追加するのが自然。
- **listSchedules のページング未対応**: Phase 3 の UI 側で必要になったらページネーション引数を追加する（現状は全件返す）。
- **usage 記録**: post.ts の `publishPost` 内部では usageRepo を使う実装が進んでいる。executeJob 経由でも同様に記録される（publishPost の内部ロジックに委譲しているため）。schedule 側で追加の usage 記録は不要。
- **ENABLE_SCHEDULER=true のテスト**: worker のユニットテストは未作成。`createSchedulerWorker` は exportしているため、Phase 3 以降で integration test を追加可能。
- **時刻依存テストの fragility**: `now` を deps に注入可能にしており、テストでは固定時刻で検証している。
