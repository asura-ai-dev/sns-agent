## known_gaps

- **dev サーバーの HMR 安定性**: Next.js 15.5.15 on macOS + pnpm worktree 環境で、`.next/routes-manifest.json` / `_buildManifest.js.tmp.*` に関する `ENOENT` が 2 回目以降のリクエストで散発的に発生する（webpack・turbopack 共通）。**これは本タスクの実装とは無関係な既知の dev watcher 不具合**（task 3004 handoff でも同種の記述あり）。初回リクエストでは必ず HTTP 200 が返る。production build (`next build`) は未実施
- **使用量の集計単位**: 現在の `SummaryCards` の "Est. cost" は `/api/usage/summary` が返す `estimatedCostUsd`（UsageSummary 型）をそのまま表示する。より厳密に月次境界を切る場合は、API 側で `range.from` / `range.to` を基準にしたフィルタが必要。この表示は MTD（月初〜現在）を想定しているが、実装は API レスポンスの生値を信頼する
- **アクティビティの時系列不整合**: 予約ジョブの `schedule.created` 用タイムスタンプは `scheduledAt`（未来時刻）を使うため、降順ソート後は「まだ実行されていない未来の予約」が最上部に来ることがある。これは仕様通り（運用上便利）だが、将来的に past / upcoming を分けるなら検討余地あり
- **posts API の limit**: ダッシュボードは 50 件までしか post を取得しない。月次トータルが 50 を超える場合は summary の totalPosts 表示が実際より少なくなる。AC-11 は「サマリが表示される」ことのみ要求しているため spec 違反ではないが、正確な総数が欲しくなったら `/api/posts/count` もしくは `meta.total` を使う設計拡張が必要
- **認証セッション未接続**: `lib/api.ts` は `SNS_AGENT_API_KEY` 環境変数をそのまま Bearer に使う。Web UI のセッションベース認証（Phase 6 相当）と接続されたら、サーバー側で cookies から API key / session を取り出す middleware 層を挟む必要がある
- **デモデータは未実装**: タスク指示では「デモデータ可」とあったが、現状は空配列 / ゼロ値フォールバックで empty-state UI を表示する方針を採用。デモデータを求められたら `lib/api.ts` の fallback を拡張する

## 重要判断

- **デザイン方向**: 既存の設定画面 (`/settings/accounts`, `/settings/audit`) が採用している "Operations Ledger" 系紙面デザインに合わせつつ、ダッシュボードは「Morning Edition（朝刊）」という broadsheet メタファーに拡張。Fraunces serif + DM Sans + JetBrains mono のタイポグラフィ階層、warm `#FFFDF8` surface、double-rule・dashed border・crop mark・roman numeral eyebrow といった印刷物の意匠で紙面感を強化
- **SSR icon import**: `@phosphor-icons/react/dist/ssr` 経由でインポートし Server Component から直接使えるようにした（client/server 境界を跨がない）
- **空状態 vs デモデータ**: 「API 未起動でも壊れない」こと優先で完全な empty-state UI を選択。press wire offline banner + zero-valued summary cards + "No dispatches yet" のペーパー空状態 + 各 bureau card の "connect … account" CTA の組み合わせで、データがない状態でも dashboard の骨格と次の行動が分かる
- **集約ロジックを page.tsx に閉じ込める**: コンポーネント側は「表示に必要な props の形」にのみ依存させ、DB モデル / API レスポンスの差し替えがあっても `page.tsx` の集約関数 (`aggregatePlatformStats` / `buildActivity`) のみ直せば済むようにした。各 dashboard component は server/client 中立で単体テスト可能
- **server-only パッケージは使わない**: `import "server-only"` は node_modules に直接存在しないためリスクを避け、コメントで RSC 専用であることを明記するに留めた
