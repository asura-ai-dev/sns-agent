## 完了済み項目

- `packages/cli/src/commands/usage.ts` 新規作成
  - `sns usage report --platform --range <daily|weekly|monthly> --json`
    - 既存 SDK `client.usage.report()` を呼ぶ
    - `--range` のバリデーション（daily/weekly/monthly 以外は VALID_RANGE エラー）
    - デフォルト range は monthly
    - 列定義: ID / PLATFORM / ENDPOINT / REQUESTS / SUCCESSES / FAILURES / COST_USD / PERIOD
  - `sns usage summary --json`
    - 既存 SDK `client.usage.summary()` を呼ぶ
    - JSON モードは summary オブジェクト直出力、人間可読は metric/value テーブル

- `packages/cli/src/commands/llm.ts` 新規作成
  - `sns llm route list --json`
    - 汎用 `client.get<ApiResponse<LlmRoute[]>>("/api/llm/routes")` を呼ぶ
  - `sns llm route set --platform --action --provider --model --temperature --max-tokens --fallback-provider --fallback-model`
    - `--platform` `--provider` `--model` は必須（未指定時 VALID_REQUIRED エラー）
    - `--temperature` `--max-tokens` は数値パース（不正時 VALID_NUMBER エラー）
    - `--fallback-provider` と `--fallback-model` は両方 or なしのみ許可（VALID_FALLBACK_PAIR エラー）
    - `client.post<ApiResponse<LlmRoute>>("/api/llm/routes", body)` で POST
  - `sns llm route delete <id>`
    - `client.delete<ApiResponse<{success}>>("/api/llm/routes/:id")` を呼ぶ
    - `encodeURIComponent(id)` で URL エスケープ

- `packages/cli/src/commands/skills.ts` 新規作成
  - `sns skills list --json` → `GET /api/skills`
  - `sns skills pack --platform --provider`
    - `--platform` `--provider` は必須
    - `POST /api/skills/pack` with `{ platform, provider }`
  - `sns skills enable <id>` → `POST /api/skills/:id/enable`
  - `sns skills disable <id>` → `POST /api/skills/:id/disable`
  - `sns skills show <id> --json` → `GET /api/skills/:id`

- `packages/cli/src/index.ts` に新コマンド 3 つを登録
  - import: `registerUsageCommand`, `registerLlmCommand`, `registerSkillsCommand`
  - `registerAccountsCommand(program)` の直後に登録
  - 既存構造・エラーハンドリング (`exitOverride` / `main()`) は変更せず

## 実装設計メモ

- **SDK 側の LLM / skills リソース追加は行わなかった**。理由は以下:
  - SDK には汎用 `get` / `post` / `delete` public メソッドが既に存在する
  - CLI 内部型 `LlmRoute` / `SkillPackage` を定義して型安全を維持
  - SDK に専用リソースを追加するのは llm/skills API が実装される Phase 5 側のタスクで行う方が整合的
  - CLI 側の骨格実装はこれで満たせるため、スコープ外の SDK 拡張は避けた

- `getGlobalOpts` / runCommand パターンは既存の `accounts.ts` と完全一致。スタイル統一を優先。

- usage の SDK リソース (`client.usage.report` / `client.usage.summary`) は task 3001 以前に既に実装済みだった。今回はそれを呼ぶだけ。

- API 未実装時の挙動: `client.get` が 404 を返すと `SdkError` が throw され、`runCommand` の catch で `normalizeError` 経由で `Not found: ...` と整形される。フォーマッター (json/table) が自動選択されるため JSON モードではエラーオブジェクトとして出力される。

- Phase 4/5 の API 実装が完了するまで、llm/skills の実通信は 404 を返す。これは仕様通り（チケット指示「API 未実装時はエラーメッセージ表示」）。

## 重要判断

- **SDK 拡張は見送り**: スコープ外のため、CLI から汎用 `client.get/post/delete` を直接呼ぶ方針を取った。
- **バリデーションは CLI 側で実施**: `--platform` 必須チェック、`--range` 値チェック、fallback ペアチェックは API を呼ぶ前に実施し、APIエラーではなく VALID\_\* エラーコードで返す（UX 向上）。
- **encodeURIComponent** を id パラメータに適用: ユーザーが不正な文字を含む id を渡した場合の URL インジェクション防止。
