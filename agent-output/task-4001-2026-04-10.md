## known_gaps

- `LineInboxStore` の実装は usecase 層側の追加タスクが必要（Phase 6 の Inbox チケットで行う想定）。現状の provider-line は空ストアを DI される前提で動くので、Webhook で蓄積したデータを実際に listThreads/getMessages から返すには、外部で LineInboxStore 実装を作り `LineProvider` に注入する必要がある
- Flex Message (リッチメッセージ) は拡張ポイント (`extra.messages`) として通路だけ用意。v1 では標準ヘルパは追加していない（チケット仕様上、v1 では対応不要）
- LINE の Channel Access Token 発行時に、現状は JWK(json) ではなく「呼び出し側が JWK → PKCS#8 PEM 変換を済ませた状態」を期待する設計。JWK 直接入力のサポートは次段階で検討可能
- PublishPostInput の core 型に platform 固有の `extra` フィールドは存在しないので、`LinePublishMode`/targetId/targetIds は `(input as unknown as { extra })` で抽出している。将来的に core 側に provider-specific ペイロードのインターフェースを追加することを推奨
- `refreshToken(accountId)` は引数を使わず常に constructor の oauth から再発行する。複数 LINE チャンネルを 1 プロセスで扱う場合は provider を複数インスタンス化する運用を想定
- `publishPost` の broadcast は承認フロー（Phase 6）との連携を前提とするが、本チケットでは provider 層での承認チェックは実装していない（承認は usecase 層で行う設計）

## 重要判断メモ

- **deletePost 戦略**: 仕様どおり `throw new ProviderError(...)` を選択（返却値 `{ success:false }` ではなく throw）。core/usecase 層は PROVIDER_ERROR として扱う。テストは `rejects.toBeInstanceOf(ProviderError)` で確認済み
- **credentials の parser 共通化**: `post.ts` / `inbox.ts` / `webhook.ts` で重複を避けるため `credentials.ts` に集約
- **署名検証での raw body 要件**: Hono の `c.req.text()` で raw body を取得してから provider に渡している（既存の Instagram ルートと同じパターン）
- **ProviderRegistry 登録のフェイルセーフ**: LINE 環境変数が揃っていない場合も provider 登録をスキップし警告のみ（既存 X/Instagram と整合）
- **JWT 実装**: 依存を増やさないため node:crypto のみで RS256 を実装。`jsonwebtoken` や `jose` は導入していない
- **connectAccount の OAuth ダイアログ**: LINE はユーザー OAuth ではなくチャンネル設定 + JWT のため、`authorizationUrl` は返さず直接 account を返す設計。OAuth ダイアログが必要な LINE Login（別製品）は本チケットのスコープ外
