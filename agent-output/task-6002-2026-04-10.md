### API - 承認ルート

- `apps/api/src/routes/approvals.ts` を本実装に置き換え
  - `GET /api/approvals` (RBAC: `approval:review`)
    - クエリ: `status` (pending/approved/rejected/expired、デフォルト pending)、`resourceType`、`requestedBy`、`page`、`limit`
    - meta に `pendingCount` を含める（Header バッジで使用）
    - 日時フィールドは ISO 文字列でシリアライズ
  - `GET /api/approvals/pending-count` (RBAC: `approval:review`): `{ data: { pendingCount } }` のみを返す軽量エンドポイント
  - `POST /api/approvals/:id/approve` (RBAC: `approval:manage`)
    - 他ワークスペースの request を誤操作しないよう事前に所有権チェック（404 返却）
    - `approveRequest` を呼び executor 実行
    - レスポンス: `{ data: { request, executorMissing } }`
  - `POST /api/approvals/:id/reject` (RBAC: `approval:manage`)
    - ボディ: `{ reason?: string }`（空ボディ許容）
    - `rejectRequest` 実行後、resourceType==="post" なら対象 Post の status を `draft` に戻す（承認待ち解除）
  - 依存注入:
    - `buildPostDeps`: DrizzlePostRepository + DrizzleAccountRepository + providers + encryptionKey
    - `buildApprovalDeps`: ApprovalRepository + AuditLogRepository + executors
    - `postExecutor`: 承認時に post の status が `scheduled` であれば draft に戻してから `publishPost` を呼ぶ（publishPost は draft 限定のため）

### API - 投稿フロー統合

- `apps/api/src/routes/posts.ts` の `POST /api/posts/:id/publish` を更新
  - 事前に `getPost` で所有権確認、status が draft 以外なら ValidationError
  - `requiresApproval("post:publish", actor.role, { platform })` を呼び判定
  - 承認必要な場合:
    - `createApprovalRequest` で承認リクエスト作成
    - Post の status を `scheduled`（承認待ち）に更新
    - **202 Accepted** + `{ data: Post, meta: { requiresApproval: true, approvalId } }` を返す
  - 承認不要な場合（editor/admin/owner）:
    - 従来通り `publishPost` を呼び、200 + `{ data: Post, meta: { requiresApproval: false } }`
  - `requiresApproval` / `createApprovalRequest` / `DrizzleApprovalRepository` / `DrizzleAuditLogRepository` / `ApprovalUsecaseDeps` を import に追加

## known_gaps

- **expireStaleRequests の定期実行はスケジューラ未配線**。usecase と repository は用意したが、cron/job ランナーからの呼び出しは将来タスク（承認フロー単体では保留でよい）。
- **Post 以外の resourceType（line_broadcast / budget_exceed）の executor 未登録**。現在 executors マップには `post` のみ登録。line broadcast と budget 超過承認は resourceType のみ扱えるが、実行系は未接続。broadcast と budget は別チケット（Task 4xxx / 受信系）の完成後に executor を追加する想定。
- **requestedBy の agent/user 判定が文字列 prefix ベース**。本来は `resolveActorByUserId` / `resolveActorByApiKey` で引き直すべき。UI 側 (`ApprovalListItem.formatActor`) はヒューリスティックで、API は `actor.type === "agent"` を使用している。
- **監査ミドルウェアとの二重記録の可能性**。auditMiddleware が `/api/approvals/*` の POST を自動記録しつつ、usecase 側でも `approval.approve` などを記録するため、同一操作が 2 件出る。監査ログは追記のみなので重複は害ではないが、次回のクリーンアップ対象。
- **i18n 未対応**。UI のラベル（"Writ of Approval", "Approve", "Reject", "Filed", "Your seal is final…"）は英語ハードコード。既存 UI の audit ページも同方針（"Operations Ledger"）なので揃えたが、i18n 導入時にまとめて置換が必要。
- **Post 詳細プレビュー取得** (`ApprovalDialog`) は `/api/posts/:id` を直接叩く別 fetch。SWR/キャッシュ無しのため dialog 開くたびにリクエストが走る。優先度低。
- **ブラウザ実機確認は未実施**。Next.js ビルドは通っているが、DOM 操作（dropdown 開閉、stamp animation）は agent-browser 等で視覚検証していない。ui-visual/ui-interact は evaluator フェーズで担当予定。
- **migrations の適用**: `approval_requests` テーブルは既存 schema に定義済みだが、新規環境の migration 実行は Orchestrator 側で。
