## known_gaps

- **認証のダミー実装**: web UI からの `/api/audit` 呼び出しは `credentials: "include"` のみで実際のセッション Cookie 認証は Task 2001 の簡易ヘッダ方式（`X-Session-User-Id`）に依存する。本物のセッション Cookie 認証は後続タスクで実装される想定
- **DB クエリ最適化**: countByWorkspace は `count(*)` を別クエリで発行している。大量データで遅くなる場合は将来 over() window 関数や計上 snapshot に置き換える余地がある
- **CSV エクスポート**: v1 スコープでは JSON のみ。UI には「export」ボタンがあり JSON を落とす。CSV は v1.5 以降
- **estimated_cost_usd の自動入力**: 現時点で auditMiddleware は cost を `null` で記録する。LLM や SNS API 呼び出しの実コストは usage-records 側で別途追跡されており、audit log との関連付けは後続の改善課題
- **リクエストボディのダブル読み取り**: auditMiddleware が `c.req.raw.clone()` で body を読むため、ハンドラ側も `c.req.json()` を使う場合は Hono のキャッシュ機構に依存する。現時点の routes 実装では問題ないが、ストリーミングボディには未対応
- **監査ログの AUDIT ログ**: `/api/audit` 自身の読み取りはエクスパンドパスとして除外している（infinite loop 防止）。Owner の audit:read 閲覧履歴を記録したい場合は別のメカニズムが必要
- **UI のサーバーサイドレンダリング**: page.tsx は "use client" のため /settings/audit はクライアントサイドでフェッチする。初期レンダリングが JS 待ちになる点は許容（管理画面のため）
