## 既知のギャップ

- `POST /api/posts`・`PATCH /api/posts/:id` のレスポンスは従来どおり `serializePost` (socialAccount/schedule なし)。一覧のみ enrich する設計とした。詳細取得 (`GET /api/posts/:id`) の enrich は Task 2006 の scope 外なので次フェーズの判断に委ねる
- `countByWorkspace` と `findByWorkspace` は 2 クエリで total + data を取得する。将来、Drizzle の window 関数等で 1 クエリ化できるが v1 は可読性優先
- SQLite の `LIKE` はデフォルトで大文字小文字を区別しない (ASCII 範囲)。PostgreSQL に切り替える際は `ILIKE` 化が必要になる。drizzle でのポータブルな書き方は v1.1 で検討
- 既存の `apps/api` には vitest 環境が用意されていないため、HTTP レイヤ（Hono ルート）の E2E テストは書いていない。統合テストは core ユースケース層で行った。routes/posts.ts のクエリパラメータ解釈は TypeScript の型チェックで保証されており、evaluate フェーズで E2E を追加する余地あり

## 重要判断

- **`listPosts` の返り型を `Post[]` から `PostListItem[]` に変更**: design.md が要求する `socialAccount` / `schedule` フィールドをレスポンス整形層ではなく usecase 層で埋めることで、API / CLI / SDK いずれの経路からも統一された形状を返せるようにした（非機能要件「API レスポンス整形の一貫性」）。`PostListItem extends Post` としたため既存コードの型互換性は維持
- **`PostUsecaseDeps.scheduledJobRepo` を optional にした**: 既存 mock や `apps/api/src/routes/approvals.ts` 内の `buildPostDeps` を壊さないため。scheduledJobRepo 未注入時は `schedule: null` が返る（後方互換）
- **クエリパラメータの複数値**: カンマ区切り (`?platform=x,line`) と同一キー繰り返し (`?platform=x&platform=line`) の両方をサポート。CLI からは前者が、Web UI の URLSearchParams 系コードは後者が出やすいため
- **limit の上限**: チケット指示「max 100」を厳守。usecase 側で `Math.min(limit, 100)`、API 側で `> 100` なら ValidationError の二段構え（早期エラー化）
- **from/to の semantics**: `created_at` 基準。design.md にも「date range」と書かれているだけなので created_at を採用。`published_at` 基準にする場合は別途 `orderBy=publishedAt` と組み合わせる運用で対応可
