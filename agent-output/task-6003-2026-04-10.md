## known_gaps

- **承認フロー本体との統合**: `sendReply` は現状「呼ばれたら送る」最小構成。spec 上「承認フローを経由する」とあるが、`packages/core/src/usecases/approval.ts` は汎用 ApprovalExecutor 設計なので、将来的に `sendInboxReply` 呼び出し前に `requiresApproval("inbox:reply", actor.role, ...)` を評価し、承認必須のケースでは approval_requests を作ってから deferred execute する拡張が残っている。現状は RBAC (editor+) と監査ログミドルウェア (auditMiddleware) で最低限のガードを持つ。
- **X の webhook 署名検証**: `/api/webhooks/x` は X Account Activity API 相当の body 形状を受理する最小実装で、CRC challenge や signature 検証は未実装 (Phase 4 で provider-x の webhook が完成したら差し替え)。
- **既読管理**: conversation_threads にはスキーマ上 `unread_count` 等がないため、UI では表示されない。v1 スコープ外。
- **Webhook → UI リアルタイム反映**: 現状はユーザーが手動で「更新」ボタンを押すかスレッドを再選択する必要がある。SSE / polling での自動更新は v1 スコープ外。
- **Provider 側の sendReply 実装**: `provider-x` は `sendReply` 未実装、`provider-line`/`provider-instagram` はそれぞれ `packages/provider-*/src/inbox.ts` が担当 (このタスクでは確認のみで未触)。sendReply が provider に無い場合は ProviderError が投げられる (テスト済み)。
- **Web 全体ビルド (turbo)**: `pnpm build` の `@sns-agent/web` タスク最終段階で `routes-manifest.json` を `standalone` へ copy する処理が ENOENT で失敗する pre-existing な infrastructure 問題を確認 (stash して差分なし状態でも再現)。このタスク起因ではない。Next.js 単体の `next build` は正常完了。
