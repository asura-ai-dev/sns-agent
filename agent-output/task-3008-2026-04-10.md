## known_gaps

1. **Users / AgentIdentities の API 未実装**: `/api/users`, `/api/users/invite`, `/api/users/:id/role`, `/api/agent-identities` は Phase 2 時点で未実装のため、UI は 404 / ネットワークエラーを検出した場合 localStorage のデモデータへフォールバックする。API 実装後は自動的にライブへ切り替わる（`usingDemoData` フラグが false になる）。後続タスクで API を追加する際、レスポンス形式は `{ data: Row[] }` / 作成系は `{ data: Row }` を期待する。
2. **認証連携**: 現在のロール判定は URL クエリ / localStorage ベース。Phase 3 末で認証ミドルウェアが完成した際に `useCurrentRole` を server session 経由に差し替える必要がある。
3. **OAuth コールバック後の一覧再読込**: `window.open` で別タブを開く実装のため、認証完了後は手動 refresh を促すメッセージを表示している。`postMessage` でオリジンタブを起こす仕組みは Phase 4 以降で検討。
4. **ロール変更 API の楽観的更新**: 現状は state を先に更新してから API を叩く実装。API 失敗時のロールバック処理は未実装。API 実装後にエラー時は旧ロールへ戻す分岐を追加すること。
5. **エージェント ID の削除 / ローテーション**: 作成のみサポート。削除・キー再発行は本チケットスコープ外。
6. **LLM ルーティング / 予算ポリシー**: SettingsShell にはナビエントリを disabled で配置済み。Phase 4/5 で `settings/llm/page.tsx` / `settings/budget/page.tsx` を追加し、disabled を外す想定。
7. **テスト自動化**: vitest ユニットテストは本チケット未追加。`RoleBadge` の色マッピング / `maskKey` / `timeUntil` など pure 関数は将来テスト追加の候補。

## 重要判断

- **既存デザイン言語（Operations Ledger）の踏襲**: audit 画面で確立した Fraunces + DM Sans + 紙面 + 暖色 + 罫線のテイストを設定画面全体に展開した。独自の「Settings Atelier」方針でゼロから作るのではなく統一感を優先。これにより Phase 3 フェーズ内の画面遷移でユーザー体験が揃う。
- **SDK ではなく fetch を直接使用**: audit 画面の先例に合わせ、Web UI は `SnsAgentClient` ではなく `fetch("/api/...")` を直接使う。SDK は CLI / テストから使う位置付け。
- **Users/Agents の API 未実装問題**: チケットのスコープは UI 実装であり、API 未実装を理由に UI を作らないのは不適切。graceful fallback としてデモデータを用意し、API 実装後は自動的にライブに切り替わる設計にした。`usingDemoData` バナーを出すことで隠蔽を避けている。
- **`/settings` の扱い**: サイドバー構成を左ナビで表現する SettingsShell を採用したので、`/settings` 単体ページは `redirect('/settings/accounts')` とした。チケットの「デフォルト: アカウント接続」要件を満たす。
- **ConfirmDialog の role 変更確認**: RBAC 上クリティカルな操作なので、ドロップダウン変更即時 commit ではなく確認ダイアログを挟む方針にした。
