# Task 2005: 予約投稿 - スケジューラとジョブ管理

## 実装指示

1. **予約ユースケース** `core/usecases/schedule.ts` を実装する
   - `schedulePost(postId, scheduledAt)`: 投稿を予約する
     - 投稿の status を 'scheduled' に更新する
     - `scheduled_jobs` テーブルにジョブを作成する（status: 'pending'）
     - `scheduledAt` が過去の場合は ValidationError
   - `updateSchedule(jobId, scheduledAt)`: 予約日時を変更する。status が 'pending' でなければエラー
   - `cancelSchedule(jobId)`: 予約をキャンセルする。投稿の status を 'draft' に戻す
   - `listSchedules(workspaceId, filters)`: 予約一覧取得
   - `getSchedule(jobId)`: 予約詳細取得
   - `executeJob(jobId)`: ジョブを実行する（ワーカーから呼ばれる）
     - ジョブを lock する（楽観的ロック: status='pending' AND locked_at IS NULL）
     - 投稿の即時公開（publishPost ユースケース）を呼ぶ
     - 成功: status を 'succeeded' に更新
     - 失敗: attempt_count をインクリメント、max_attempts 未満なら 'retrying' + next_retry_at を設定、到達なら 'failed'

2. **Polling ワーカー** `apps/api/src/worker/scheduler.ts` を実装する
   - 10 秒間隔で `scheduled_jobs` テーブルを polling する
   - 条件: `status = 'pending' AND scheduled_at <= now()` または `status = 'retrying' AND next_retry_at <= now()`
   - デッドロック回避: `locked_at` が 5 分以上前のジョブは再取得可能にする
   - ジョブ実行は `executeJob` ユースケースに委譲する

3. **再試行ロジック**
   - exponential backoff: 30s, 120s, 480s
   - `next_retry_at = now() + backoff[attempt_count - 1]`
   - max_attempts（デフォルト 3）到達で最終失敗

4. **予約 API エンドポイント** `routes/schedules.ts` を実装する
   - `GET /api/schedules` - listSchedules（RBAC: post:read）
   - `POST /api/schedules` - schedulePost（RBAC: post:publish）。ボディ: `{ postId, scheduledAt }`
   - `GET /api/schedules/:id` - getSchedule（RBAC: post:read）
   - `PATCH /api/schedules/:id` - updateSchedule（RBAC: post:publish）
   - `DELETE /api/schedules/:id` - cancelSchedule（RBAC: post:publish）

5. **ワーカー起動**
   - `apps/api/src/index.ts` でサーバー起動と同時にワーカーも開始する
   - 環境変数 `ENABLE_SCHEDULER=true` の場合のみワーカーを起動する（テスト時は無効化可能）
