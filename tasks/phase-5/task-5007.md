# Task 5007: OpenAI Codex / ChatGPT サブスク認証 provider 追加

## 目的

`OPENAI_API_KEY` を使う既存 `openai` provider とは別に、`openai-codex` provider を追加し、
ChatGPT / Codex 系の認証情報で LLM を呼べる土台を実装する。

## 実装指示

1. **provider 追加**
   - `openai` と別に `openai-codex` provider を追加する
   - 既存 `openai` API キー方式は壊さない

2. **credentials 保存**
   - workspace 単位で `openai-codex` の認証情報を保存できるようにする
   - access token / refresh token / expiresAt / status を扱える形にする
   - 保存時は既存の秘密情報保存方針に合わせて暗号化する

3. **OAuth 接続 API**
   - 接続開始、callback、status、disconnect の API を追加する
   - callback 成功後に credentials を保存する
   - token refresh が必要な場合に再認証導線を作れるようにする

4. **LLM adapter**
   - `packages/llm/src/adapters/openai-codex.ts` を追加する
   - `LlmAdapter` インターフェースに従い、ChatGPT / Codex 系認証情報で chat を実行する
   - token 期限切れ時は refresh または失敗状態を返せるようにする

5. **Agent Gateway 統合**
   - route の provider が `openai-codex` のとき、DB の credentials を参照して adapter を解決する
   - route の provider が `openai` のときは、従来通り `OPENAI_API_KEY` を使う

6. **Web UI**
   - LLM ルーティング画面で provider に `openai-codex` を追加する
   - 接続状態を表示する
     - 未接続
     - 接続済み
     - 期限切れ
     - 再認証必要
   - 接続 / 再接続 / 切断ボタンを用意する

## 実装ヒント

- `openai-codex` は既存 `openai` の置き換えではなく別 provider として扱う
- adapter の構築元を `env 固定` ではなく `provider ごとの resolver` に分ける
- UI 文言は非エンジニア向けにし、`APIキー方式` と `ChatGPT / Codex ログイン方式` の違いが分かるようにする
- `wire offline` と `認証切れ` は意味が違うため、表示メッセージを分ける

## 非目標

- `openai` provider の API キー経路を削除すること
- ChatGPT の web session / cookie を直接使う非公式な実装
- OpenAI 以外の subscription auth を同時に追加すること

## 注意点

- 公式仕様が変わる可能性があるので、着手時に OpenAI 側の最新 docs を必ず確認する
- 認証エラーと API 障害は UI 上で区別できるようにする
- 既存 Chat UI / Agent Gateway / Skills 実行基盤との整合を崩さない
