# Task 6001: 監査ログ

## 目的

全操作の監査ログを記録・参照する機能を実装する。追記のみ（UPDATE/DELETE 不可）のログテーブルに actor、操作、対象、結果、コストを記録し、管理画面とCLIから検索・参照できるようにする。

## 実装指示

1. **AuditLogRepository 実装**
   - `create(log)`: ログ挿入（INSERT のみ）
   - `findByWorkspace(workspaceId, filters)`: フィルタ付き一覧取得
     - filters: actorId, actorType, action, resourceType, platform, from, to
     - ページネーション対応
   - **UPDATE / DELETE メソッドは定義しない**（追記のみの強制）

2. **監査ログユースケース** `core/usecases/audit.ts`
   - `recordAudit(input)`: 監査ログを記録する
     - input: workspaceId, actorId, actorType, action, resourceType, resourceId, platform, socialAccountId, inputSummary, resultSummary, estimatedCostUsd, requestId
   - `listAuditLogs(workspaceId, filters)`: フィルタ付き一覧
   - `exportAuditLogs(workspaceId, filters, format)`: CSV 等のエクスポート（v1 では JSON のみ）

3. **自動監査記録ミドルウェア** `middleware/audit.ts`
   - 書き込み系エンドポイント（POST, PATCH, DELETE）の完了後に自動で監査ログを記録する
   - context から actor 情報を取得し、リクエスト・レスポンスの概要を input_summary / result_summary に記録する
   - request_id はリクエスト ID ミドルウェアが付与した値を使う
   - 機密情報（トークン、パスワード等）は input_summary に含めない

4. **監査ログ API** `routes/audit.ts`
   - `GET /api/audit` - 監査ログ一覧（RBAC: audit:read -> admin 以上）
     - クエリパラメータ: actorId, actorType, action, resourceType, platform, from, to, page, limit
     - レスポンス: `{ data: AuditLog[], meta: { page, limit, total } }`

5. **監査ログ画面** `settings/audit/page.tsx`
   - フィルタバー: Actor 検索、アクション種別、プラットフォーム、日付範囲
   - ログテーブル
     - 各行: 日時、Actor（名前 + バッジ user/agent/system）、アクション、対象リソース、SNS、結果（成功/失敗）、コスト
   - 行クリックで詳細モーダル（input_summary, result_summary の JSON 表示）
   - ページネーション
